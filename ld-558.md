## 项目概述

**环保与可持续生活平台（GreenLife）** — 面向城市居民的碳中和与绿色生活数据可视化大屏系统。通过实时模拟数据展示碳排放趋势、垃圾分类达标率、回收积分排行和公益项目进度，引导公众关注环境保护与可持续发展。

**项目类型：数据可视化大屏（纯前端）**

## 项目主要需求

### 一、核心数据模型

1. **碳足迹 CarbonFootprint**：字段包括 id、userId、category（交通/饮食/用电/用水/购物）、amount、unit、date、source。支持记录个人碳排放数据、按月统计汇总、生成月度报告、年度对比分析。此模型贯穿全栈：DB → model → store → 组件，所有碳排放相关页面均消费此模型。

2. **垃圾分类 WasteStats**：字段包括 date、category（可回收/有害/湿垃圾/干垃圾）、weight、location、collectorId。支持记录每日各区域垃圾分类数据、统计各区域达标率、社区排行。此模型贯穿全栈：DB → model → store → 组件，垃圾分类分析页面消费此模型。

3. **回收积分 RecyclingCredit**：字段包括 userId、item、weight、points、redeemedAt、exchangeItem。支持积分兑换、回收记录、积分排行榜。此模型贯穿全栈：DB → model → store → 组件，回收积分大屏消费此模型。

4. **公益项目 CharityProject**：字段包括 id、name、description、targetAmount、currentAmount、participants、startDate、endDate、status。支持项目进度追踪、捐赠人排行、参与趋势分析。此模型贯穿全栈：DB → model → store → 组件，公益项目页面消费此模型。

### 二、核心页面（≥ 4 个）

1. **`/` 碳中和主屏**：展示全球/城市碳排放实时数据仪表盘，左侧区域碳排放热力图，中间碳排放趋势折线图，右侧个人碳足迹分类饼图与减排目标进度条。消费 CarbonFootprint 数据。

2. **`/waste` 垃圾分类分析屏**：各区域垃圾分类达标率柱状图、月度趋势折线图、TOP10 社区排行列表、四分类占比环形图。消费 WasteStats 数据。

3. **`/recycling` 回收积分大屏**：积分排行榜（动态滚动）、兑换趋势面积图、回收物品分类统计雷达图、实时积分流水。消费 RecyclingCredit 数据。

4. **`/charity` 公益项目屏**：项目进度条展示、捐赠人排行榜、参与趋势折线图、项目状态分布饼图。消费 CharityProject 数据。

5. **`/personal` 个人碳档案**：个人碳足迹月度趋势折线图、各分类占比环形图、减排建议卡片、成就徽章墙。消费 CarbonFootprint + RecyclingCredit 数据。

### 三、共用组件

- `CarbonGauge`：碳排放仪表盘（被碳中和主屏、个人碳档案使用）
- `WastePieChart`：垃圾分类环形图（被垃圾分类分析屏、碳中和主屏使用）
- `RecyclingBarChart`：回收积分柱状图（被回收积分大屏使用）
- `CharityProgressChart`：公益项目进度图（被公益项目屏使用）
- `TrendLine`：通用趋势折线图（被所有页面使用）
- `HeatMap`：热力图（被碳中和主屏使用）
- `DataCard`：数据卡片（被所有页面使用）
- `RankList`：排行榜组件（被回收积分大屏、公益项目屏使用）
- `AnimatedNumber`：数字动画组件（被所有数据卡片使用）
- `ScreenAdapter`：大屏自适应容器（被所有页面使用）

### 四、横切关注点

1. **大屏适配（1920x1080 基准）**：全局使用 rem/vw 适配方案 + ECharts 自适应 resize。ScreenAdapter 组件监听窗口变化，统一计算缩放比例。此关注点触达全局样式和每个图表组件。

2. **数据模拟服务**：使用 mock 数据生成器模拟实时数据推送。每个数据模型有独立的 mock 生成器文件，通过定时器模拟数据变化，WebSocket 连接管理器统一管理连接生命周期。此关注点触达 mock 层和 WebSocket 连接管理。

## 代码架构要求（强制执行）

### 枚举必须单独维护为常量文件

所有枚举类型必须在 `src/constants/enums.ts` 中集中定义，业务代码中**严禁硬编码**枚举值字符串。README 中必须列出所有枚举的定义位置和使用位置对照表。

**共享枚举定义**：
- `CarbonCategory`：TRANSPORT / DIET / ELECTRICITY / WATER / SHOPPING
- `WasteType`：RECYCLABLE / HAZARDOUS / WET / DRY
- `ProjectStatus`：UPCOMING / ACTIVE / COMPLETED

### 全局异常处理

前端请求拦截器统一错误提示：在 `src/utils/request.ts` 中封装 Axios 实例，配置响应拦截器拦截 HTTP 错误和业务错误码，统一调用 `ElMessage.error()` 展示错误提示，禁止各页面自行 catch 后直接 alert。

### 操作日志

关键写操作（碳足迹记录、积分兑换、公益项目捐赠）通过审计工具函数记录操作日志，在 `src/utils/audit-log.ts` 中封装，记录操作人、操作类型、操作时间、操作详情。

### 代码严禁合并职责到单一文件

- 每个功能模块必须拆分为独立文件。例如：碳足迹模块拆分为 `carbonStore.ts`（状态管理）、`carbon.d.ts`（类型定义）、`CarbonGauge.vue`（图表组件）、`useCarbonData.ts`（组合式函数）。
- 鼓励细粒度拆分，单个文件不超过 300 行。
- 前端组件按功能拆分，每个页面拆分为多个子组件。
- 按实体/功能拆分为 store、types、constants、components、hooks、mock、utils 等独立文件。

### 核心实体贯穿全栈

每个核心实体必须完整贯穿：**类型定义（types/*.d.ts）→ 状态管理（stores/*.ts）→ 组合式函数（hooks/*.ts）→ 页面组件（pages/*.vue）→ 图表组件（components/charts/*.vue）**。禁止在页面中直接操作原始数据，必须通过 store 中转。
